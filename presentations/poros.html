Why POROs?

“Programs must be written for people to read, and only incidentally for machines to execute.” — Harold Abelson

POROs are conventions POROs are abstractions

Rules for POROs

  • Don’t insert code that doesn’t belong
  • Avoid inheritance, including Module inclusion with includes
  • Treat them as leaves of complexity

Models

What do they do

  • Couple data into an easy to understand representation of that data
  • Make it easy to pass that data around
  • Make it easy to reuse that data in different parts of code
  • Make it easy to transform date from a single source of truth

Models

CreditCard = Struct.new(:number, :cvv, :expiration_date) do
  # Assume parameters from the client of the form:
  #   "creditCard": {
  #     "number": "9999-9999-9999-9999",
  #     "cvv": "123",
  #     "expMonth:" 9,
  #     "expYear": 27,
  #   }
  def self.from_params(params)
    expiration_date = Date.new(params["expYear"] + 2000, params["expMonth"], 1)

    new(params["number"], params["cvv"], expiration_date)
  end

  def is_visa?
    number.starts_with?("4")
  end

  def is_mastercard?
    number.starts_with?("5")
  end
end

Models

  • I like to just put these in /app/models even though they aren’t DB backed
  • Try and keep these pure
  • Use class methods as constructors
  • Use public methods as representations of the data
  • AKA Value Objects

Forms

What do they do

  • Parameter validation
  • Parameter data sanitizing
  • Turning parameters into useful objects

Forms

  puts "hey"

Forms

  • You may have heard about these before, but my use case is simpler
  • Try and keep these pure too
  • Other names could be Parameters, Inputs
  • Consider using include ActiveModel::Validations in Forms to get a nice validation convention for free

Services

What do they do

  • A generic place for complicated business logic
  • Helps keeps controllers and models from exploding in size
  • Isolates complexity
  • Help business logic become composable
  • Makes up for Ruby’s lax module loading system

Services

Refactor Example

Services

  • Services are going to be an entrypoint into a lot of other code. They are nodes of complexity; not leaves.
  • Expect these classes to get a little hairy with a lot of business logic
  • Use the other POROs to whittle down at this complexity

Connections

What do they do

  • Provide an interface for a third party API
  • Transform an API’s calls into methods that encapsulate your application’s logic
  • Give a convention for how devs should make remote requests

Connections

Refactor Example

Connections

  • Use a Connection superclass to abstract out request generalities
  • Also a useful abstraction outside of Rails

Queries

What do they do

  • Encapsulate complicated database queries
  • Give you room to break out of an ORM to do logic in code or in SQL
  • Since all pure SQL lives in one place, you know where to look before DB upgrades or transitions
